# 崧達營運作業系統長期方向

> 決策日期：2026-07-14
> 狀態：公司長期產品與系統調整方向
> 適用範圍：`quote-system` 的規劃、資料建模、自動化、介面、指標與排程
> 維護原則：具體數字以正式資料與當次盤點為準；本文件保存方向、邊界與驗收原則，不保存憑證。

## 一、決策摘要

公司目前有四條營運主線：

1. 業務開發決策數據化、情資化、自動化。
2. 設備與產品技術支援資料規模化、客戶體驗優化、排程可控化。
3. 行政工作自動化。
4. 行銷事務與業務協作同步化。

系統不做一次性重建。現有客戶、報價、訂單、促銷、拜訪、漏斗、工單、設備、活動、素材與監控能力繼續運作；改造方式是補上共同資料與流程底座，再逐條形成閉環。

四條主線分成：

- 核心引擎：業務成長、技術服務。
- 賦能引擎：行政自動化、行銷業務協作。

最終每位使用者登入後，系統要能回答：

1. 今天最該處理什麼？
2. 為什麼是這件事？
3. 完成後，系統會自動推進什麼？

## 二、共同營運閉環

所有新能力優先落入同一閉環：

`訊號 → 判斷 → 工作項 → 負責人 → 期限/SLA → 處理結果 → 指標/下一個事件`

不得只新增一張無人負責、無期限、無結果回寫的資訊頁。儀表板若不能支援決策或產生後續工作，不應列為優先開發。

## 三、共同資料與系統底座

### 1. 統一識別與關聯

- 客戶、聯絡人、設備、產品/SKU、工單、拜訪、商機、活動、報價與訂單使用穩定識別值關聯。
- 不以名稱比對作為長期跨領域關聯方式。
- 保留既有 Notion 正式資料；新增關聯前先盤點欄位、影響筆數與樣本，資料寫入遵守 dry-run 與 read-back。

### 2. Append-only 營運事件帳本

統一描述可影響流程的事件，例如：

- 客戶建立、認領、階段改變。
- 拜訪完成、要求報價、同意試用、提及競品。
- 報價送出、核准、拒絕、訂單成立或取消。
- 活動報名、到場、素材接觸、名單交接。
- 設備安裝、保養、報修、派工、完修或重開。

同一業務事件不論來自網頁、LINE、批次匯入或排程，都必須走相同的領域規則，不能因入口不同產生不同結果。

### 3. 統一工作項與排程模型

共同欄位至少包含：來源事件、工作類型、負責人、優先級、狀態、建立時間、期限、SLA、處理結果及關聯客戶/設備/活動。

業務跟進、技服派工、行政例外與行銷交接可有不同狀態機，但應能進入同一個人工作佇列與主管例外視圖。

### 4. 可重算的指標層

- 儀表板不可長期依賴請求時跨多個 Notion DB 即時扇出。
- 指標定義必須有分子、分母、期間、時區、排除條件與資料來源。
- 歷史事實應可追溯、重算與對帳；快取不是歷史資料庫。
- Notion可繼續作為正式資料及操作介面，另以適合的關聯式資料/指標層承接事件與分析，不做大爆炸式搬遷。

### 5. 自動化控制面

所有正式自動化逐步具備：執行履歷、輸入摘要、成功/失敗狀態、冪等鍵、重試、補跑、逾時、告警與人工接管。排程不得只有隱藏在 workflow 檔案中的時間設定而無營運可見性。

## 四、四條主線

### A. 業務開發：從看漏斗到產生下一步

保留現有開發漏斗、商機偵測、拜訪建議、追蹤名單及成交回寫，逐步增加：

- 可解釋的客戶機會分數：需求訊號、設備狀態、購買週期、競品、拜訪反應、活動參與等。
- Next Best Action：建議客戶、商品、行動與原因；先規則化並可人工回饋，再評估模型化。
- 自動養成：逾期未聯絡、報價未回覆、活動後未跟進時產生工作。
- 風險預警：商機停滯、重要客戶流失、價格或競品風險。
- 主管決策：新增機會、階段轉換、停滯原因、預測營收及可信度。

核心指標：首聯時間、各階段轉換率、成交率、銷售週期、預測準確率、有效機會數。具體目標值須在基線盤點後決定。

### B. 技術支援：從工單紀錄到服務履約

保留既有工單與設備頁，逐步增加：

- 工單狀態機：待判斷、待排程、已排程、處理中、待零件、待客戶、完成、重開；實際選項先以現有 Notion schema 盤點為準。
- 派工看板：工程師技能、區域、工時、交通、零件與可用時段。
- SLA：回覆、到場與結案期限，逾時升級且保留人工調整權。
- 設備履歷：安裝、保養、故障、零件、更換與使用年限。
- 技術知識：故障碼、處理步驟、產品文件與已驗證解法；結案時沉澱、發布前審核。
- 客戶入口：案件進度、補充照片/資訊、可選時段、完工確認與滿意度。
- 主動服務：依設備年齡、維修頻率與保養週期提出預防性工作，不直接無人審核對外通知。

核心指標：首次回應時間、準時到場率、一次修復率、平均結案時間、重開率、工程師產能、客戶滿意度。

### C. 行政：從逐件處理到例外管理

- 正常案件依規則自動驗證與流轉，行政聚焦資料缺漏、規則衝突、失敗任務與逾期例外。
- 報價、折扣、訂單與文件沿用既有伺服器端驗證與簽核狀態機，不得為自動化繞過。
- LINE、Excel、表單及批次資料先進驗證/預覽流程，再寫正式資料。
- 建立行政工作台：待簽核、資料缺漏、重複資料、失敗任務、逾期事項與執行履歷。
- 後續再依實際量體評估應收、發票、採購、庫存及費用流程，未盤點前不預設欄位或系統邊界。

核心指標：無人工處理率、平均處理時間、錯誤率、退件率、逾期件數、節省工時。

### D. 行銷與業務：以 Campaign 統一協作

- Campaign 統一連結目標客群、商品、素材、活動、優惠、負責人與期間。
- 每位客戶保留來源與互動軌跡；名單交給業務時建立負責人、期限及建議行動。
- 業務回饋有興趣、時機未到、產品不符、使用競品等標準結果，回流行銷調整客群、商品與素材。
- 報價/訂單回寫 Campaign，區分直接來源、受影響與無法確認，不以最後接觸粗暴取代完整歸因。
- 素材追蹤使用情境與結果，但不蒐集無法產生決策的虛榮指標。

核心指標：名單接受率、24/48小時跟進率、活動到商機轉換率、受影響營收、Campaign毛利、素材轉換表現。

## 五、分期路線圖

### 0–30天：先修定義與斷鏈

1. 盤點統一識別、relation、入口副作用與產品權威來源。
2. 定義事件格式、工作項最小模型與四條主線 KPI 字典。
3. 建立修改前回歸基線，優先修正入口不一致與跨域斷鏈。
4. 不新增會大量寫正式資料的自動化。

### 31–90天：交付第一個閉環

1. 個人工作佇列及主管例外視圖。
2. 可解釋的機會分數與 Next Best Action 第一版。
3. 技服狀態機、SLA及基本派工看板。
4. Campaign交接與成交歸因。
5. 行政例外工作台與自動化執行履歷。

### 3–6個月：規模化

1. 客戶自助服務與審核式技術知識庫。
2. 預防性保養、設備風險與商機停滯預警。
3. 穩定高量行政流程逐批自動化。
4. 行銷營收歸因及規則式培育；資料量與品質足夠後才導入預測模型。

## 六、不破壞既有功能的交付鐵則

1. 先盤點再修改：記錄現有資料欄位、API response、畫面流程與排程行為。
2. 相容性優先：新增欄位先 optional、讀取支援舊資料、寫入採雙軌或 feature flag；完成回填與驗證前不得硬切。
3. 每批只解一個可驗證問題；結構重構與行為改變分開。
4. 不直接改寫正式 Notion 大量資料；一律 dry-run、前20筆樣本、影響筆數、使用者核准、寫後 read-back。
5. 程式修改至少通過 `npx tsc --noEmit`，再實跑目標 API/UI 正常案例與防禦案例。
6. 報價、訂單、價格快照、促銷驗證、權限、Webhook簽章與醫事監控資料安全屬不可退步區。
7. 自動化先 shadow mode：只計算/記錄建議，不自動寫正式資料或對外通知；準確度與例外穩定後再逐步開放。
8. 每批保留回滾路徑、變更前後指標與錯誤告警；無法證明未破壞既有流程時不得部署。

## 七、當前盤點優先級

依 2026-07-14 初步程式盤點，先驗證下列風險，不直接假定結論：

1. 客戶跨訂單、工單、設備與拜訪的識別/Relation是否一致。
2. 同一拜訪由 REST、LINE或批次匯入時，是否觸發相同後續規則。
3. 工單是否有完整更新、派工、SLA與排程衝突控制。
4. 產品 catalog 與 Notion enrichment 的權威邊界是否符合既有欄位擁有權。
5. 決策儀表板是否缺乏可追溯歷史指標，或因即時扇出而不穩定。
6. GitHub Actions與cron是否缺少補跑、重試、告警及營運執行履歷。

盤點結果應另存具日期的文件；本文件只在公司方向、系統邊界或交付鐵則改變時更新。
