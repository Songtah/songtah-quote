# 崧達系統敏感資訊分級與存取規範

版本：2026-07-20  
狀態：全系統改造強制規格；正式套用角色矩陣前須先完成帳號影響盤點

## 1. 核心原則

1. 預設拒絕：沒有明確權限就不提供敏感資料，不以「前端沒有顯示」代替 API 授權。
2. 最小揭露：同一 API 依角色只回傳完成工作所需欄位；列表、搜尋、匯出與 AI 輸入遵守相同規則。
3. 欄位級控管：模組 view/edit 只能作第一層閘道，高敏感欄位另做 field policy。
4. 資料範圍控管：一般業務原則上只看本人負責客戶與本人／被指派的工作；跨區、跨人與全公司統計另授權。
5. 匯出另授權：能在頁面查看不等於能批次匯出；下載、列印、PDF、複製名單與 AI 分析必須個別控管。
6. 全程稽核：高敏感資料的檢視、匯出、分享、修改、權限異動與失敗嘗試都留下可追溯紀錄。
7. 密鑰永不下放：密碼、token、secret、cookie、Authorization header、私鑰不得出現在前端、API body、稽核明文或錯誤訊息。
8. 對外連結最小化：公開分享只包含交付目的需要的資料，採不可猜測識別碼、有效性控制與存取稽核；不得暴露內部備註、成本、毛利或稽核資訊。

## 2. 四級資料分類

| 等級 | 名稱 | 定義 | 代表資料 | 基本控制 |
|---|---|---|---|---|
| L1 | 公開 | 經公司核准可對外提供 | 公開型錄內容、已核准品牌素材、正式報價分享中的必要商務內容 | 可公開讀取；仍需防竄改與版本管理 |
| L2 | 內部 | 僅供公司工作使用，外流影響有限 | 一般產品內部資訊、活動排程、去識別化統計、一般作業狀態 | 必須登入；依模組 view 控制；禁止任意公開索引 |
| L3 | 機密 | 可識別客戶、員工或商務條件，外流會造成業務／個資風險 | 客戶電話地址統編、拜訪內容、負責業務、報價、訂單、設備、工單、活動報名、區域名單 | 角色＋資料範圍＋欄位政策；匯出另授權；檢視與異動稽核 |
| L4 | 嚴格機密 | 可造成重大財務、安全、權限或競爭風險 | 密碼雜湊、token/secret、帳號權限、成本毛利、全公司業績排行、高風險稽核、批次客戶名單、系統設定 | 預設拒絕；指定管理角色；必要時遮罩；完整稽核；不得送入外部 AI 或公開分享 |

## 3. 推薦角色存取矩陣

符號：`本人`＝本人負責／被指派範圍；`團隊`＝經理管理範圍；`全`＝全公司；`遮`＝遮罩或只顯示摘要；`—`＝拒絕。

| 資料／能力 | 業務 | 業務經理 | 行政 | 中央管理 | 總經理／admin | 外部客戶 |
|---|---:|---:|---:|---:|---:|---:|
| 客戶基本資料與聯絡方式（L3） | 本人 | 團隊 | 工作需要 | 全 | 全 | — |
| 拜訪內容／下一步（L3） | 本人 | 團隊 | 遮 | 全 | 全 | — |
| 報價與訂單（L3） | 本人 | 團隊 | 工作需要 | 全 | 全 | 自己收到的核准內容 |
| 售價與促銷規則（L2/L3） | 工作需要 | 團隊 | 工作需要 | 全 | 全 | 核准報價內價格 |
| 成本、毛利、特殊底價（L4） | — | 遮／另授權 | 工作需要／另授權 | 全 | 全 | — |
| 個人業績（L3） | 本人 | 團隊 | 遮 | 全 | 全 | — |
| 全公司排行與經營指標（L4） | — | 團隊 | 遮 | 全 | 全 | — |
| 設備與工單（L3） | 本人／指派 | 團隊 | 工作需要 | 全 | 全 | — |
| 活動報名個資（L3） | 本人／指派 | 團隊 | 工作需要 | 全 | 全 | 本人確認資訊 |
| 客戶名單批次匯出（L4） | —／另授權 | 團隊／另授權 | 工作需要／另授權 | 全 | 全 | — |
| 帳號與權限（L4） | — | — | 遮 | 管理 | 全 | — |
| 操作稽核與風險事件（L4） | 本人事件摘要 | 團隊摘要 | 遮 | 全 | 全 | — |
| 密碼、token、secret（L4） | — | — | — | —（只可輪替，不可讀回） | —（只可輪替，不可讀回） | — |

> 「業務經理」目前若未在帳號模型中明確存在，不得用名稱、部門字串或前端判斷暫代；須先建立可驗證角色或資料範圍。

## 4. 各通道強制要求

### 頁面與元件

- 敏感欄位由伺服器授權結果決定是否回傳；前端不可先收到完整資料再以 CSS 或條件隱藏。
- L3/L4 列表預設不展示完整電話、地址、統編與內部備註；工作需要時才揭露。
- L4 數值不可透過 HTML、React props、data attribute 或 hydration payload 偷渡給未授權角色。
- 管理畫面顯示資料等級與匯出限制，敏感操作需明確二次確認。

### API

- 所有一般 API 使用 `withApiAuth` 宣告模組與 action；高敏感 API 再疊加 field policy 與 scope policy。
- 不接受前端傳入的 salesperson、owner、role 作為授權依據；資料範圍由 session 與伺服器規則計算。
- 回應 DTO 依角色裁切欄位，禁止把 Notion page 或完整領域物件直接序列化回前端。
- 未授權回 401/403；不存在與無權查看可視風險選擇一致錯誤，避免枚舉 ID。
- PDF、CSV、批次名單與 AI 分析端點視為 export，另做授權與稽核。

### 分享與 PDF

- `/share/[id]` 與 `/api/quotes/[id]/pdf` 保留既有 URL 契約，但只提供核准交付欄位。
- 永不包含成本、毛利、內部簽核意見、操作紀錄、客戶內部備註、其他客戶或員工資訊。
- 分享識別碼須不可猜測；新增有效／撤銷控制前，既有分享 URL 不得直接失效。
- 分享存取記錄不得寫入 query secret、cookie 或完整敏感內容。

### 匯出與 AI

- 匯出權限與 view 分離；記錄操作者、時間、條件、資料等級與筆數，不在 audit 保存整份內容。
- L4 原則上不得送入外部 AI；L3 必須先最小化、移除不必要識別資料並記錄用途。
- Prompt、錯誤紀錄與模型除錯輸出不得包含 token、secret、密碼或無關客戶資料。

### 稽核與紀錄

- 稽核遮罩鍵名至少涵蓋 password、token、secret、authorization、cookie、apiKey、privateKey、credential。
- L3/L4 的批次檢視、匯出、分享、價格／折扣例外、權限異動、資料刪除與拒絕事件必須稽核。
- 稽核不可儲存完整 L4 原值；必要時保存欄位名、雜湊、前後狀態摘要與關聯 ID。

## 5. 實施順序與切換閘門

### Phase A：唯讀盤點

1. 列出帳號角色、accountType、模組權限與資料範圍；只顯示帳號識別與權限，不讀回密碼。
2. 盤點各 API 的 auth rule、回應 DTO、匯出能力及敏感欄位。
3. 建立目前合法工作案例，確保安全強化不誤擋正常訂單、報價、拜訪及服務流程。

### Phase B：安全底座

1. 建立 `DataClassification`、field policy、scope policy 與 export policy 的伺服器端 helper。
2. 擴充 audit 遮罩鍵名與高風險事件類型。
3. 先保護帳號、權限、成本毛利、稽核、批次匯出與 AI，再處理一般 L3 欄位。

### Phase C：逐模組切換

1. 每次只切一個模組，先 shadow/dry-run 比對允許與拒絕結果。
2. 同時測合法案例不誤擋、非法案例確實被擋。
3. 實際驗證頁面、API status/body、匯出與 audit read-back 後才部署下一批。

### Phase D：舊權限收斂

1. 所有正式帳號都有明確 permissions 與資料範圍後，才能移除「無 permissions 視為全開」相容行為。
2. 切換前列出影響帳號與前 20 筆樣本，取得使用者確認。
3. 保留可回復的分批開關；不得用前端 workaround 暫時放寬 L3/L4。

## 6. 目前已知差距

- 現有模型以模組 view/edit 為主，尚無欄位分級、資料範圍與匯出權限。
- `lib/permissions.ts` 對沒有 explicit permissions 的舊帳號採一般模組全開，以維持相容；在帳號盤點完成前不可直接改成全面拒絕。
- `lib/notion/permissions-model.ts` 僅對 accounts/admin 欄位缺失預設拒絕，其他模組仍有相容性開放預設。
- `lib/audit.ts` 已針對常見敏感鍵名及字串中的 Bearer／Authorization／Cookie／密鑰指派做遮罩；仍需建立自動化回歸測試與欄位 allowlist，持續防範未知命名格式。
- 公開分享、PDF、AI、批次名單與匯出須逐項確認是否使用最小 DTO；目前不得假設已完全符合本規範。

## 7. 待管理者確認的業務決策

1. 業務經理角色如何識別，以及管理的業務範圍從哪個權威資料取得。
2. 一般業務能否查看非本人負責客戶的聯絡方式，以及陌生開發池的揭露規則。
3. 行政可查看的成本、毛利、業績與拜訪內容範圍。
4. 客戶名單、報價與訂單的匯出角色及核准流程。
5. 公開報價分享的有效期限、撤銷方式與既有永久連結相容政策。
